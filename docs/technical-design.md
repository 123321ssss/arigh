# Technical Design

## 1. Product Positioning

This project is an internal-team AI workspace built around two explicit roles: `admin` and `member`.

- The public homepage `/` is indexable and can be discovered through search.
- Access to the actual workspace remains gated.
- Members register only with invite codes generated by admins.
- After first registration, all users sign in with `email + password`.
- The member area focuses on conversations, prompt templates, short-task agents, and per-message model switching.
- The admin area focuses on invite-code lifecycle, account state, model registry, budgets, usage, and audit events.

The system no longer relies on magic links, SMTP delivery, or invitation emails as the primary onboarding path.

## 2. Authentication And Access

### 2.1 Public Entry Points

- `/` is public and indexable.
- `/login` is the password sign-in page and should be `noindex`.
- `/register` is the invite-code registration page and should be `noindex`.
- `/setup` is a one-time bootstrap page for the first admin account and should be `noindex`.
- `/app` and `/admin/*` are always authenticated.

### 2.2 Bootstrap Flow

The first administrator is created through `/setup`.

- `/setup` is accessible only when no admin account exists.
- If an admin already exists, the page is blocked unless a deployment bootstrap secret is provided.
- The bootstrap API creates a Supabase user with:
  - `email_confirm: true`
  - `app_metadata.app_role = "admin"`
  - `app_metadata.active = true`
- The API then signs that user in server-side and redirects to `/admin`.

Environment:

- `APP_BOOTSTRAP_SECRET` is optional.
- When set, `/setup?token=...` can be reopened deliberately.

### 2.3 Registration Flow

Registration is invite-only.

1. Admin generates an invite code in `/admin/members`.
2. Admin copies either the raw code or a register link like `/register?invite=...`.
3. Member opens `/register`.
4. Member submits:
   - invite code
   - display name
   - email
   - password
5. Server validates invite code rules.
6. Server creates a Supabase user with:
   - `app_metadata.app_role`
   - `app_metadata.active = true`
   - invite metadata
7. Server consumes one invite-code use.
8. Server signs the user in and redirects to `/app` or `/admin`.

### 2.4 Login Flow

Login is server-mediated password login.

1. Browser calls `POST /api/auth/password-login`.
2. Route handler uses Supabase SSR server client to sign in with password.
3. Session cookies are written server-side.
4. Local profile is created or updated from Supabase metadata.
5. Inactive users are immediately signed out and blocked.

This keeps the browser from directly depending on third-party auth SDK calls for the primary sign-in path.

## 3. Invite-Code Lifecycle

Invite codes replace email invitations.

### 3.1 Data Model

Core fields:

- `id`
- `codeHash`
- `encryptedCode`
- `codePreview`
- `role`
- `status`
- `maxUses`
- `usedCount`
- `allowedEmailDomain`
- `note`
- `createdBy`
- `expiresAt`
- `lastUsedAt`
- `createdAt`

Security model:

- Raw invite code is generated once.
- `codeHash` is used for lookup and validation.
- `encryptedCode` is retained so admins can still copy the code later from the admin UI.
- `codePreview` is safe to show in tables.

### 3.2 Status Rules

Supported statuses:

- `active`
- `revoked`
- `expired`
- `exhausted`

Rules:

- Expired codes are rejected once `expiresAt < now`.
- Exhausted codes are rejected once `usedCount >= maxUses`.
- Revoked codes are rejected immediately.
- Optional email-domain restriction is enforced during registration.

### 3.3 Admin Operations

Admin actions:

- create invite code
- copy raw code
- copy registration link
- revoke invite code
- inspect usage count and expiry

Routes:

- `POST /api/admin/invite-codes`
- `PATCH /api/admin/invite-codes/[id]`

## 4. Persistence Strategy

### 4.1 Repository Mode

The repository layer now supports two modes:

- Database-first mode when `DATABASE_URL` is configured and runtime is not `test`
- In-memory fallback when no database URL is available

Database-backed entities:

- profiles
- invite codes
- conversations
- messages
- agent runs
- model configs
- prompt templates
- tool policies
- usage events
- audit logs

### 4.2 Current Deployment Reality

The code is ready to persist through Postgres via Drizzle, but actual persistence still depends on a real `DATABASE_URL`.

If `DATABASE_URL` is empty:

- the app remains usable
- invite codes, members, conversations, and messages fall back to in-memory state
- serverless restarts can lose those records

To enable durable storage:

1. set `DATABASE_URL`
2. run `npm run db:push`
3. set the same `DATABASE_URL` in deployment environment variables

### 4.3 Schema Highlights

The schema includes:

- `invite_codes`
- `profiles`
- `conversations`
- `messages`
- `agent_runs`
- `model_configs`
- `prompt_templates`
- `tool_policies`
- `usage_events`
- `audit_logs`

Conversation schema now stores both:

- `default_model_key`
- `last_used_model_key`

Messages store model metadata in encrypted payload plus `payload_meta`.

## 5. Chat And Model Switching

### 5.1 Model Registry

Admins maintain the model catalog in `/admin/models`.

Each model includes:

- `key`
- `label`
- `providerModelId`
- `enabled`
- `description`
- `defaultSystemPrompt`
- `promptPrefix`
- `monthlyBudgetUsd`
- input/output price data

Only enabled models are exposed to members in the composer.

### 5.2 Per-Message Model Switching

Model switching is explicit and scoped to the next send.

Behavior:

- Composer always shows a model select control.
- Selected model is sent as `modelKey` in `POST /api/chat/stream`.
- The selected model only affects the next request.
- The conversation is not locked to one model.

### 5.3 Conversation Model Semantics

Conversation summary tracks both:

- `defaultModelKey`: the model selected when the conversation was created
- `lastUsedModelKey`: the actual most recent model used in that thread

The rail UI shows both so admins and members can distinguish the original default from the latest usage.

### 5.4 Message Model Semantics

Assistant messages store metadata:

- `modelKey`
- `modelLabel`
- `promptTemplateId`

This allows the thread UI to show a model badge on each assistant response, even when a single thread mixes different GPT models.

## 6. Chat Lifecycle

`POST /api/chat/stream` executes the following sequence:

1. authenticate current user
2. load conversation and selected model
3. validate model availability
4. run budget guard against usage summary
5. save the latest user-side snapshot
6. run either:
   - local mock stream, or
   - real OpenAI-compatible stream via Vercel AI SDK
7. collect tool steps into `agent_runs`
8. attach model metadata to the resulting assistant message
9. persist final conversation snapshot
10. persist usage event
11. append audit log

Tool policy remains:

- `calculator`
- `clock`
- `http_fetch_whitelist`
- `prompt_template_lookup`

Agent bounds remain:

- `stopWhen: stepCountIs(4)`
- `maxRetries: 1`

## 7. Frontend Composition

The UI follows composition-first React patterns rather than boolean prop branching.

Examples:

- `AccessShell.Root / Stage / Card`
- `AppShell.Root / Sidebar / Main / Header / Aside`
- `ChatComposer.Provider / Frame / InputField / Controls`
- explicit page variants:
  - login page
  - register page
  - setup page
  - member workspace
  - admin workspace

The design language remains “editorial control room”:

- warm paper background
- charcoal panels
- teal accent
- serif display headings
- non-symmetric layout

## 9. Testing Strategy

### 9.1 Authentication

Required checks:

- first admin can be created through `/setup`
- `/setup` closes after admin exists
- valid invite code can register a member
- invalid / expired / revoked / exhausted invite codes are rejected
- registered member can log in with email and password
- inactive member is blocked after login attempt

### 9.2 Invite-Code Admin Flow

Required checks:

- admin can generate invite codes
- register link contains invite code
- admin can revoke active invite codes
- member cannot call admin invite-code APIs

### 9.3 Chat And Models

Required checks:

- same conversation can send two messages with different models
- usage is recorded against the actual model used
- assistant messages display model badge after refresh
- disabled models are not usable

### 9.4 Build Safety

Production build must succeed without SMTP or magic-link configuration.

## 10. Operational Notes

- SMTP is no longer required for the primary onboarding flow.
- Supabase is still used for session and credential storage.
- For a real persistent deployment, `DATABASE_URL` is mandatory.
- For Vercel deployment, environment variables must include:
  - `APP_URL`
  - `APP_ENCRYPTION_SECRET`
  - `NEXT_PUBLIC_SUPABASE_URL`
  - `NEXT_PUBLIC_SUPABASE_ANON_KEY`
  - `SUPABASE_SERVICE_ROLE_KEY`
  - `DATABASE_URL` for durable persistence
  - optional `APP_BOOTSTRAP_SECRET`
